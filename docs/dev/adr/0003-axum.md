# ADR-0003: axum for the HTTP layer

**Status:** Accepted  
**Date:** 2025

## Context

We needed an async HTTP framework to serve the SPARQL 1.1 Protocol, Graph Store
HTTP Protocol, and the web UI. The main candidates in the Rust async ecosystem are:

- **axum** — built on `tower` + `hyper`; composable middleware via `tower::Service`;
  maintained by the Tokio team
- **actix-web** — actor-based; historically the fastest in benchmarks; separate runtime
  from Tokio
- **warp** — filter-based composition; built on `hyper`; less active maintenance

## Decision

Use **axum** with the **tokio** runtime.

## Rationale

- **Tokio alignment:** The rest of the async stack (file I/O, shutdown signals) already
  uses Tokio. axum is a first-party Tokio project, so they share the same runtime
  without bridging overhead.
- **tower middleware:** Authentication, CORS, and timeout middleware are all standard
  `tower::Layer` implementations. This means we can compose them without framework-specific
  glue code, and swap individual layers without touching the application logic.
- **`Arc<RwLock<Datastore>>`** state sharing: axum's `State` extractor is designed for
  this pattern — shared, cloneable state per request. The datastore is read-locked for
  queries and write-locked for SPARQL Update operations.
- **Maintenance:** axum is actively maintained and tracks Tokio and hyper releases closely.

## Trade-offs

- actix-web tends to score higher in raw throughput benchmarks due to its actor model.
  For an RDF triplestore, query execution time dominates HTTP handling time, so this
  difference is unlikely to matter in practice.
- axum's `tower::Service` trait is more complex to extend than actix-web's
  `FromRequest` / `Responder` traits.
