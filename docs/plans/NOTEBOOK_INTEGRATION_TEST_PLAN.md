# Plan: Headless Notebook Integration Test + Datalog Cell Fix

> Tracked under [#28 Jupyter kernel epic](https://github.com/daghovland/rdf-datalog/issues/28).

## Background

Manual testing of `notebooks/dagalog_intro.ipynb` against `dagalog-kernel` surfaced
two already-fixed protocol bugs (Jupyter `root_dir` defaulting to `notebooks/`,
and a premature `idle` status that broke strict clients like `nbclient`), plus
one open issue: the `%%datalog` cell fails to parse.

## Decision: bare-triple syntax vs. bracket syntax

The failing cell body is:

```
%%datalog
?x <http://example.com/colleague> ?y :-
    ?x <http://example.com/worksFor> ?org ,
    ?y <http://example.com/worksFor> ?org .
```

This uses a bare/space-separated `subject predicate object` triple form.
Investigation of `datalog_parser/src/lib.rs`, the original DagSemTools ANTLR
grammar (`Datalog.g4`), and every existing `.datalog` fixture confirmed this
form **was never valid** in either the C# original or the Rust port — only
`[s, p, o]` (bracket) and `p[s, o]` (predicate-first) atoms are supported.
`docs/plans/JUPYTER_KERNEL_PLAN.md`'s own `%%datalog` example also (incorrectly)
used bare-triple syntax, so this was a documentation/example bug, not a one-off
notebook typo.

**Decided fix (user choice): rewrite the notebook cell and the kernel plan's
example to use the already-supported bracket syntax.** No changes to
`datalog_parser`.

Corrected cell body:

```
%%datalog
[?x, <http://example.com/colleague>, ?y] :-
    [?x, <http://example.com/worksFor>, ?org],
    [?y, <http://example.com/worksFor>, ?org] .
```

## Deliverable 1: content fix (green phase, no parser changes)

- `notebooks/dagalog_intro.ipynb` — rewrite `step6-datalog` cell source to the
  bracket form above; outputs (including the stale `step1-turtle` "Loaded 0
  triples." left over from the pre-fix broken run) get regenerated by actually

- `docs/plans/JUPYTER_KERNEL_PLAN.md` — rewrite the `%%datalog` example (lines
  ~87–89) to bracket syntax.

## Deliverable 2: Rust-native headless notebook integration test

Goal: drive the real `dagalog-kernel` binary over the actual Jupyter wire
protocol (ZMQ + HMAC-signed JSON), no Python/Jupyter dependency, asserting on
captured output per cell — fulfilling "run headless notebooks to confirm
output from example cells."

### Infrastructure change: expose a `[lib]` target

`dagalog-kernel` is currently `[[bin]]`-only, so its `protocol.rs` (message
framing/HMAC) isn't reusable by an external test binary. Add:

```toml
[lib]
name = "dagalog_kernel"
path = "src/lib.rs"
```

`src/lib.rs` declares `pub mod {cell, output, protocol, session, sockets}`;
`main.rs` becomes a thin entry point using `dagalog_kernel::sockets::run_kernel`.
Pure refactor, no behavior change — done now, not gated on test review, since
nothing about it is testable in isolation.

### Test client design (`dagalog-kernel/tests/support/mod.rs`)

- `KernelHarness::start(repo_root: &Path) -> Self`
  - picks 5 free ports via `TcpListener::bind("127.0.0.1:0")` + drop
  - writes a connection file (JSON) to a temp path with a random HMAC key
  - spawns `env!("CARGO_BIN_EXE_dagalog-kernel") launch --connection-file <path>`
    with `current_dir(repo_root)` (mirrors `--ServerApp.root_dir=.`)
  - connects a `zeromq::DealerSocket` to `shell` and a `zeromq::SubSocket`
    (empty topic filter) to `iopub`, reusing `dagalog_kernel::protocol` for
    encode/decode — same wire code the kernel itself uses, so the test can't
    pass by agreeing with a buggy duplicate implementation
  - "nudge" handshake: repeatedly send `kernel_info_request` (bounded retries
    with a short overall timeout) until both a shell reply and at least one
    iopub message are observed, working around ZMQ's PUB/SUB slow-joiner race
- `KernelHarness::execute(&mut self, code: &str) -> ExecuteOutcome`
  - sends `execute_request`, collects iopub messages for that `msg_id` until
    `status: idle`, plus the `execute_reply` on shell
  - `ExecuteOutcome { status, stream: Option<String>, error: Option<String>, rich: Option<HashMap<mime, content>> }`
- `KernelHarness::shutdown(&mut self)` — `shutdown_request`, then kill the
  child process if still alive (also on `Drop`, so a panicking test doesn't
  leak kernel processes)

Stubbed with `todo!()` bodies for the red phase — compiles, not yet functional.

### Tests (`dagalog-kernel/tests/notebook_e2e.rs`), easiest first

All `#[ignore]`d pending review; unignored one at a time in green phase.

1. `test_kernel_info_handshake` — `start()` succeeds (handshake completes) and
   `shutdown()` returns cleanly. Validates the ZMQ/protocol plumbing alone.
2. `test_turtle_cell_loads_ten_triples` — runs the notebook's `step1-turtle`
   body, asserts `status == "ok"` and `stream == Some("Loaded 10 triples.")`
   (4 triples for Alice + 4 for Bob + 2 for Acme).
3. `test_sparql_select_cell_returns_two_rows` — turtle cell then `step2-sparql`
   body, asserts `rich["text/plain"] == "2 result(s)."` and
   `rich["text/html"]` contains both `Alice` and `Bob`.
4. `test_rml_cell_loads_six_triples` — turtle cell then `step3-rml` body
   (`%%rml tests/testdata/rml_persons_mapping.ttl`), asserts
   `stream == Some("Loaded 6 triples.")` (2 CSV rows × 3 triples: type, name, age).
5. `test_reason_cell_adds_zero_triples` — turtle + rml cells then `step4-reason`,
   asserts `stream == Some("Reasoning complete. 0 triples added.")` — the
   dataset has no TBox axioms (no `rdfs:domain`/`range`, `owl:equivalentClass`,
   etc.), so no OWL 2 RL rule has a premise to fire.
6. `test_datalog_bracket_cell_materializes_colleagues` — turtle cell then the
   **corrected** bracket-syntax `step6-datalog` body, asserts
   `stream == Some("Applied 1 rule.")`, then runs `step6-query` and asserts
   `rich["text/plain"] == "2 result(s)."` (Alice↔Bob colleague pairs; the rule
   also derives the two reflexive pairs, filtered out by the query's
   `?person != ?colleague`).
7. `test_full_notebook_replay` — reads `notebooks/dagalog_intro.ipynb` directly,
   extracts all code cells in order, executes them sequentially against one
   kernel session, and asserts the exact sequence of expected outputs above
   end-to-end. This is the literal "replay the example notebook headlessly"
   test; 1–6 exist mainly to localize failures if it fails.

### Out of scope for this plan

- Updating the notebook's *stored* output cells to match fresh results (cosmetic;
  can be done by actually re-running the notebook in Jupyter once the kernel is
  fixed, not required for the test to pass).
- `%%validate` (SHACL) — not yet implemented in the kernel, not used by this
  notebook.
