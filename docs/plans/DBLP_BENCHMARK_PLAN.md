# DBLP benchmark setup — plan

## Goal

Reproduce the SPARQL query suite behind QLever's public DBLP evaluation
(https://qlever.dev/evaluation/www/#/comparison?kb=dblp) against dagalog, to
find where dagalog's SPARQL engine is missing functionality or
disproportionately slow.

This is a **diagnostic benchmark for dagalog**, not a literal head-to-head
number comparison with QLever. QLever's published DBLP numbers are measured
on the full ~1.3 billion-triple dataset using its own disk-backed,
compressed-vocabulary index (~20 GB RAM, ~25 GB on-disk index). dagalog's
`dag_rdf::Datastore` is a flat in-memory structure (`HashMap`/`Vec`-based,
confirmed via `dag_rdf/src/lib.rs`) with no disk persistence, and the
existing performance suite (`tests/performance.rs`) already shows OOM risk
at ~1.7M triples (Gene Ontology). Loading the real dataset is not feasible.
We instead load a bounded sample and use the same query suite to find
relative weaknesses in dagalog itself (missing builtins, slow query shapes),
which is what the user actually wants ("compare where improvements can be
made").

## What the benchmark actually is

QLever's evaluation page is generated by `ad-freiburg/sparqloscope`, a
generic SPARQL benchmark generator. The exact, ready-to-use DBLP query set
is checked into that repo at `benchmarks/dblp.benchmark.tsv` — 105 queries
covering joins (2-way/3-way, star/chain shapes, varying selectivity),
OPTIONAL/MINUS/FILTER EXISTS (mirrored across the same join shapes), UNION,
multi-column joins, GROUP BY (with COUNT/SUM/MIN/MAX/AVG/SAMPLE/GROUP_CONCAT,
including one query with a nested subquery), DISTINCT, transitive property
paths (`+`), regex/string functions, numeric functions, date functions, and
plain dataset statistics (triple/subject/predicate/object/literal/blank-node
counts).

Format: a 2-column TSV, one query per line —
`<name [description]>\t<full SPARQL query, single line>`. Almost every query
wraps its pattern in `SELECT (COUNT(*) AS ?count) WHERE { ... }` (sparqloscope
deliberately avoids large result transfers), except the `result-size-*` and a
few others.

This file will be committed at `tests/testdata/dblp.benchmark.tsv` — it's
~25 KB of reference data, not a large generated fixture, so it can live in
the repo directly (unlike the dataset itself).

## Dataset: sample, not full DBLP

- Source: `https://dblp.org/rdf/dblp.nt.gz` — the **N-Triples** release of
  DBLP's main bibliography (no citations). Confirmed via `curl -I` (~5.1 GB
  compressed) and by streaming the first few lines: genuine one-triple-per-line
  N-Triples, safe to truncate at any line boundary.
  - Note: `https://dblp.org/rdf/dblp.ttl.gz` (~2.26 GB) is **not** safe to
    truncate by line — it's pretty-printed Turtle with multi-line
    subject blocks (`;`-separated predicate-object lists, nested `[...]`
    blank nodes), so an arbitrary line cut produces invalid Turtle. This is
    why the N-Triples release is used instead, exactly mirroring the
    existing `wikidata-sample.nt` approach in `scripts/download_test_ontologies.sh`.
  - Citations are not needed: every predicate referenced across all 105
    benchmark queries (`dblp:formerStreamTitle`, `dblp:hasSignature`,
    `dblp:numberOfCreators`, `dblp:yearOfPublication`, `dblp:signatureOrdinal`,
    etc., plus `rdf:`/`rdfs:`/`owl:`/`terms:` vocabulary) belongs to the main
    bibliography KG, confirmed by grepping the benchmark file. No `cito:`
    (citation) predicates appear.
- Sample size: first **15,000,000 lines** (≈15M triples) of the decompressed
  stream, written to `tests/testdata/dblp-sample.nt`. Chosen as the midpoint
  of the 10–20M range. Not committed (large generated fixture, same as
  `wikidata-sample.nt`).
- Add a new section to `scripts/download_test_ontologies.sh`, modeled
  directly on the existing Wikidata section (stream → gunzip → `head -n N`,
  `set +o pipefail` around the truncating pipe, sanity-check resulting line
  count, skip if already present).

## Known gaps this will surface (found during planning, not yet fixed)

Grepping `sparql_parser/src/execute.rs`'s `eval_function_value` /
`eval_function_bool` match arms shows the only builtins currently
implemented are: `STR`, `LANG`, `STRLEN`, `DATATYPE`, `BOUND`, `REGEX`,
`LANGMATCHES`, `ISIRI`/`ISURI`, `ISBLANK`, `ISLITERAL`. The benchmark uses
several functions that fall through to the `_ => None` arm and are **not**
implemented: `STRBEFORE`, `STRAFTER`, `STRSTARTS`, `STRENDS`, `CONTAINS`,
`YEAR`, `MONTH`, `DAY`, `ABS`, `CEIL`, `FLOOR`, `ROUND`. That's ~12 of the 105
queries (`strbefore`, `strafter`, `strstarts`, `strends`,
`regex-3-contains`, `numeric-abs/ceil/floor/round`, `date-year/month/day`)
that will execute but return no/incorrect bindings until those are added.

Also: `REGEX` is currently implemented as a plain substring `contains`
check, not real regex matching (no anchors, character classes, etc.). The
benchmark's `regex-*` queries happen to use literal substrings so they may
"pass" by accident — the harness should still flag this as a correctness gap
rather than quietly relying on it.

These gaps are not fixed as part of this plan — the point of the harness is
to surface and quantify them. Follow-up work (separate, TDD'd) can pick off
each missing builtin.

## Test harness

New file `tests/dblp_benchmark.rs`, following the established
`tests/performance.rs` conventions (this repo has no shared `tests/common.rs`
— every integration test file duplicates its own small helpers, so this one
does too):

- `test_data()` / `ensure_test_data()` — same pattern as `performance.rs`.
- `dblp_load_sample` (`#[ignore]`): parses `dblp-sample.nt` via
  `turtle::parse_ntriples`, reports triple count and throughput. Asserts
  `triple_count > 1_000_000` as a basic sanity check.
- `dblp_benchmark_queries` (`#[ignore]`):
  1. Load the sample once (reuse the same `Datastore`).
  2. Parse `tests/testdata/dblp.benchmark.tsv` (split each line on the first
     `\t` into `(name_and_description, query)`).
  3. For each query: attempt `parse_query`; on failure, record `ParseFail`.
     On success, attempt `execute`; record `(elapsed_ms, row_count)` on
     success or `ExecFail` on failure.
  4. Print a results table: name | status | time (ms) | rows.
  5. Print a summary: counts of OK / ParseFail / ExecFail, total/average
     time for OK queries, and a flagged list of the known-gap queries from
     the section above (so the report makes the missing-builtins finding
     visible without needing to read full table).
  6. Loose sanity assertion only (e.g. `ok_count > 0`) — this is a diagnostic
     report, not a correctness gate; we don't yet know which queries dagalog
     can answer correctly, that's the point of running it.

No timeout handling — SPARQL execution here has no built-in cancellation, so
a pathologically slow query (e.g. one of the `join-xlarge-*` or
`transitive-path-*` shapes against unindexed nested loops) could make the
whole test run for a long time. Acceptable for an `#[ignore]`d, manually-run
diagnostic; revisit with a per-query timeout (e.g. running each in a thread
with `recv_timeout`) only if it proves to be a real problem in practice.

## Next steps after this plan

2. Add the DBLP section to `scripts/download_test_ontologies.sh`.
3. Add `tests/dblp_benchmark.rs` with the two tests stubbed, both `#[ignore]`,
   bodies doing only enough to compile (e.g. `todo!()` or a minimal
   skeleton) — for user review before implementation, per the project's TDD
   workflow.
4. After review: implement `dblp_load_sample` first (simplest), then
   `dblp_benchmark_queries`.
5. Run the harness, capture the report, and use it to decide what to fix
   next in `sparql_parser` (starting with the missing builtins listed above,
   since those are unambiguous gaps rather than performance tuning).
